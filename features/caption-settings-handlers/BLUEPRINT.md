# Blueprint: Caption settings handlers

## Architecture decisions

- **R-001 — Per-key typed commands, same shape as the existing four.**
  Follow the pattern at
  `src-tauri/src/commands/app_settings.rs:118-152` verbatim. Each new
  function is a flat `pub fn change_caption_<field>_setting(app, v)`
  with `#[tauri::command] #[specta::specta]`, reads `get_settings`,
  mutates, `write_settings`, returns `Ok(())`. No trait, no macro,
  no generic wrapper — consistency with the four existing caption
  handlers is more valuable than deduplication.
- **R-001 — Validation lives in the handler, not in a separate
  validator module.** Clamp ranges are short (one-liners) and must
  mirror `ensure_caption_defaults` exactly. Duplication is acceptable
  because `ensure_caption_defaults` still runs at settings-load time
  and is the source of truth for older on-disk files. Cite the
  `defaults.rs` ranges via code comment in each handler so drift is
  caught on review.
- **R-002 — Dispatch map, not a new hook.** The frontend already has
  one central place for setting → typed command mapping
  (`src/stores/settingsStore.ts:72-127`). Add five entries there; do
  not invent a new hook, context, or service layer. The
  `CaptionSettings.tsx` call site is unchanged.
- **R-003 — Storage shape untouched.** No migration. The flat
  `caption_*` fields remain exactly as declared in
  `src-tauri/src/settings/types.rs:295-312`.
- **R-004 — Line-cap contingency.** If
  `src-tauri/src/commands/app_settings.rs` exceeds 800 lines after the
  five handlers, move ALL nine caption handlers (the four existing +
  the five new) into a new `src-tauri/src/commands/app_settings/
  captions.rs` submodule and re-export from `mod.rs`. Do NOT
  allowlist the file.

## Single-source-of-truth placement

- Clamp ranges: two places by design —
  (a) the new typed setters (write-time defense),
  (b) `ensure_caption_defaults` (load-time defense for older files).
  Both must agree. Each new handler embeds a `// mirrors
  ensure_caption_defaults:<line>` comment so drift is caught on review.
- Storage shape: `AppSettings` in
  `src-tauri/src/settings/types.rs:295-312`, consumed verbatim by the
  frontend via specta-generated `AppSettings` in `src/bindings.ts`.
- No preview/export dual-path concern in this bundle — the flat
  `caption_*` fields are not read by the caption layout authority
  (`managers::captions::compute_caption_layout`); only
  `CaptionProfile` fields are, and those are out of scope.

## Data flow

```
UI control (CaptionSettings.tsx, unchanged)
  -> useSettings().updateSetting("caption_radius_px", 12)
  -> settingsStore.updateSetting
       - optimistic: state.settings.caption_radius_px = 12
       - dispatch: settingUpdaters["caption_radius_px"](12)  [NEW]
  -> commands.changeCaptionRadiusPxSetting(12)  [specta binding, NEW]
  -> Rust: change_caption_radius_px_setting(app, 12)        [NEW]
       - let mut s = settings::get_settings(&app)
       - s.caption_radius_px = v.min(64)
       - settings::write_settings(&app, s)
  -> settings.json on disk contains the new value
```

Read path is unchanged: `settings::get_settings` returns the whole
`AppSettings` including the five fields; the frontend's
`refreshSettings` hydrates state.

## Migration / compatibility

- No migration. The five fields are already persisted via
  `#[serde(default = …)]` defaults. Older builds that never wrote
  them produce a JSON without those keys; on load, the serde
  `default` functions fill them in, and the existing
  `ensure_caption_defaults` clamp pass normalizes anything
  out-of-range.
- If a user somehow has an existing `settings.json` with an
  out-of-range value (e.g. `caption_radius_px: 9999`), the load-time
  clamp in `ensure_caption_defaults` resets it. After this bundle,
  any write-time attempt from the frontend to set 9999 is clamped to
  64 by the handler. Both ends agree.
- No bindings field rename (the AGENTS.md one-line-per-field
  exception is not needed here — only new commands are added, not
  modified fields).
